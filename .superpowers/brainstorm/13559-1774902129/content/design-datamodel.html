Design Section 3: Data Model

How everything is stored — clean, simple, extensible

Core Tables (Supabase / PostgreSQL)

👤 users
id · name · avatar_url · email · default_currency (CAD) · created_at
👥 groups
id · name · emoji · colour · created_by · created_at
🔗 group_members
group_id · user_id · joined_at · role (admin/member)
💸 expenses
id · group_id · paid_by (user_id) · title · total_amount · currency · category · receipt_url · is_recurring · recurrence_interval · created_at · updated_at
🧾 expense_items (for receipt scanning)
id · expense_id · name · price · tax_portion · assigned_to (user_id)
⚖️ expense_splits
id · expense_id · user_id · owed_amount · split_type (equal/percent/exact/byItem/shares) · is_settled
✅ settlements
id · from_user_id · to_user_id · amount · group_id · settled_at · method (interac/cash/other)
💬 comments
id · expense_id · user_id · body · created_at
🔔 notifications
id · user_id · type · reference_id · is_read · created_at

Offline & Sync Strategy

• SwiftData mirrors all user data locally — app works 100% offline
• Changes queued locally, synced to Supabase when connection restores
• Supabase Realtime pushes live updates to all group members instantly
• Conflict resolution: last-write-wins with server timestamp
Does this data model look right? Note the expense_items table — that's what powers the per-item receipt splitting with proportional tax.
